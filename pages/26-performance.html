Bölüm 26

Performance & Optimization — FPS'i Yükselt

Profiler, Frame Debugger, GPU Instancing, SRP Batcher, Object Pool, Job System & Burst, Memory Profiler, Occlusion Culling, LOD, Async Loading — pürüzsüz 60+ FPS.

⏱ Okuma süresi: ~30 dk 🚀 15+ teknik 🎯 Zorluk: İleri
26.1

⚡ Neden Optimizasyon?

Oyun 30 FPS'in altına düştüğünde oyuncu hissedebilir. 60 FPS standart, 120/144 FPS modern hedef. VR'da 90 FPS olmazsa baş döner. Mobil'de pil ömrü = performans = düşük ısı.

Optimizasyon iki cephe:

  • CPU darboğazı — Çok script, çok fizik, çok update, garbage collection. Frame 16ms'yi geçer.
  • GPU darboğazı — Çok poligon, çok shader, çok overdraw, çok post-process. GPU işi yetiştiremez.

Önce ölç, sonra optimize. Tahmin etmek ≠ bilmek. Yanlış yeri optimize etmek vakit kaybıdır.

🎮 Gerçek Dünyadan: Genshin Impact'in Mobil Optimizasyonu

Genshin Impact Unity ile yapılan açık dünya AAA bir oyun. Aynı oyun PS5'te 4K 60FPS, alt-orta seviye Android telefonda 30FPS oynanabilir hale getirildi. Bunun arkasında: agresif LOD sistemi (uzaktaki karakter 50 poligon, yakın 50000), Occlusion Culling, dinamik gölge mesafesi, Async Scene Loading (bölgeler arası geçişte donmama), Object Pool (mermi, partikül her şey), Texture Streaming (görünmeyen texture'ları belleğe yükleme). Her teknik bu bölümde var.

26.2

📊 Unity Profiler — En Önemli Araç

Window → Analysis → Profiler. Oyun çalışırken her frame'in CPU, GPU, Memory, Rendering, Physics, Audio kullanımını gösterir.

CPU Usage Modülü

  • Timeline görünümü → Main Thread, Render Thread, Job Threads yatay barlar.
  • Her bar bir frame, içinde renkli dilimler: yeşil = Update, mavi = Render, sarı = GC.
  • Sapan barlar → o frame'de sorun olan tek metodu bul.
  • Hierarchy görünümü → metot bazında: kim ne kadar süre aldı?
  • Self ms = sadece o metod, Total ms = onun çağırdığı her şey dahil.

Profiler Açıkken Test Etmek

  1. Build Profile → Development Build ☑ + Autoconnect Profiler ☑.
  2. Build → çalıştır.
  3. Profiler penceresi otomatik bağlanır, gerçek cihazdan veri akar.
  4. Spike (ani uzama) gördüğünde Pause, o frame'i analiz et.
💡 Profiler Marker Ekleme: Kendi kodunda hangi blok ne kadar sürdüğünü görmek için: 
using Unity.Profiling;

private static readonly ProfilerMarker myMarker = new ProfilerMarker("MyAlgorithm");

private void Update()
{
    myMarker.Begin();
    RunMyAlgorithm();
    myMarker.End();
}
Profiler timeline'da "MyAlgorithm" bandı görünür.
26.3

🎬 Frame Debugger — Render Adımları

Window → Analysis → Frame Debugger. Tek bir frame'in tüm GPU draw call'larını sırayla gösterir. "Bu frame'de 1230 draw call var" — soldaki listeden tıkla, sahne sadece o ana kadar çizilmiş şekilde gözükür.

  • Enable'a bas, oyun otomatik durur.
  • Sol panelde event listesi: Clear, Shadow Pass, Opaque, Transparent, Post-process.
  • Her event'e tıkla → Scene view sadece o adımı göstersin.
  • Aynı materyalden art arda 100 draw call görüyorsan → instancing/batching kapalı.
  • Hangi mesh, hangi shader, hangi texture kullanılıyor — sağda detay.

Stats penceresi (Game view → Stats): saniye başına FPS, draw call, batched, tris, vertices, used texture memory. İlk hızlı bakış için.

26.4

📦 SRP Batcher & GPU Instancing

Draw Call = CPU'nun GPU'ya "şu mesh'i şu materyalle çiz" demesi. Her draw call CPU yükü. Azaltma teknikleri:

SRP Batcher (URP/HDRP)

  • URP Asset → Advanced → SRP Batcher ☑.
  • Aynı shader'a sahip farklı materyaller bile tek batch'te.
  • Otomatik çalışır, kod yazmaya gerek yok.
  • Şart: Shader, SRP Batcher uyumlu yazılmalı (Shader Graph hep uyumlu).

GPU Instancing

Aynı mesh'in (örn. 1000 ağaç prefab'ı) tek bir draw call ile çizilmesi:

  • Materyalde Enable GPU Instancing ☑.
  • Aynı mesh + aynı materyal + farklı position/rotation = 1 draw call.
  • Renk farklılığı için Per-Instance Properties kullan (script'ten).
  • İdeal kullanım: çimen, taş, ağaç, çakıl, askerler, partikül.

Static Batching

Sahnede sabit duran çevre objeleri (duvar, ağaç, kaya) → GameObject Inspector → Static ☑ (Batching Static). Unity build sırasında bunları büyük bir mesh'e birleştirir; ardından 1 draw call.

Dynamic Batching

Küçük (300 vertex altı) hareketli mesh'ler için runtime'da otomatik batch. Daha az verimli, modern projelerde önemi azaldı (GPU Instancing daha iyi).

26.5

♻️ Object Pool — Instantiate/Destroy Yerine

Instantiate ve Destroy pahalıdır: memory alloc, OnEnable çağrıları, garbage collector. Sürekli oluşan-yok olan objeler için (mermi, partikül, düşman, hasar yazıları) Pool kullan. 

using UnityEngine.Pool;

public class BulletSpawner : MonoBehaviour
{
    [SerializeField] private Bullet bulletPrefab;
    private IObjectPool<Bullet> pool;

    private void Awake()
    {
        pool = new ObjectPool<Bullet>(
            createFunc:  () => Instantiate(bulletPrefab),
            actionOnGet: b => b.gameObject.SetActive(true),
            actionOnRelease: b => b.gameObject.SetActive(false),
            actionOnDestroy: b => Destroy(b.gameObject),
            defaultCapacity: 20,
            maxSize: 100
        );
    }

    public void Shoot()
    {
        Bullet b = pool.Get();
        b.transform.position = transform.position;
        b.Init(pool);  // bullet kendisi pool'a iade edebilsin
    }
}

public class Bullet : MonoBehaviour
{
    private IObjectPool<Bullet> pool;

    public void Init(IObjectPool<Bullet> p)
    {
        pool = p;
        Invoke(nameof(ReturnToPool), 3f);
    }

    private void ReturnToPool() => pool.Release(this);
}

UnityEngine.Pool.ObjectPool Unity 2021+'tan beri built-in. Önceden kendi yazardık (Bölüm 25'teki Generic Pool).

📊 Pool Etkisi: 60 saniyede 1000 mermi atan bir karakterde: Instantiate/Destroy versiyonu ~25 MB garbage üretir, GC freeze'ler oluşur. Pool versiyonu 0 MB garbage, hiçbir freeze yok. AAA için olmazsa olmaz.
26.6

🔧 Job System & Burst Compiler

Bilgisayarın 8-16 çekirdeği var; Unity'nin main thread'i sadece birini kullanır. Job System ile ağır hesaplamaları diğer çekirdeklere dağıt.

Burst Compiler ise C# job kodunu native makine koduna derler — 5-50x daha hızlı. 

using Unity.Jobs;
using Unity.Collections;
using Unity.Burst;
using Unity.Mathematics;

[BurstCompile]
public struct MoveJob : IJobFor
{
    public NativeArray<float3> positions;
    public NativeArray<float3> velocities;
    public float deltaTime;

    public void Execute(int i)
    {
        positions[i] += velocities[i] * deltaTime;
    }
}

public class ParticleSwarm : MonoBehaviour
{
    private NativeArray<float3> pos, vel;

    private void Start()
    {
        pos = new NativeArray<float3>(100000, Allocator.Persistent);
        vel = new NativeArray<float3>(100000, Allocator.Persistent);
    }

    private void Update()
    {
        var job = new MoveJob
        {
            positions = pos,
            velocities = vel,
            deltaTime = Time.deltaTime
        };
        var handle = job.ScheduleParallel(pos.Length, 64, default);
        handle.Complete();
    }

    private void OnDestroy()
    {
        pos.Dispose();
        vel.Dispose();
    }
}

100.000 partikülün her frame pozisyon güncellemesi normalde 30+ ms'de yapılır; Job + Burst ile 1-2 ms. Aradaki fark: oyun donuyor mu donmuyor mu.

⚠️ Job System Kuralları: Job içinde UnityEngine sınıfları (Transform, GameObject) kullanılamaz; sadece NativeArray, NativeList, primitif tipler. Data odaklı düşün, DOTS (Data-Oriented Technology Stack)'in temelidir.
26.7

📐 LOD Groups — Uzaktakine Az Detay

Kameraya 1 metre uzaklıkta bir karakter 50000 vertex'le çizilebilir, 50 metre uzakta 500 vertex yeterlidir — fark görünmez ama GPU 100x daha az çalışır.

  1. Karakteri Blender/Maya'da 3 farklı detayda export et: LOD0 (yüksek), LOD1 (orta), LOD2 (düşük).
  2. Empty GameObject → Add Component → LOD Group.
  3. 3 LOD bölümüne 3 mesh'i sürükle.
  4. Yüzde değerlerini ayarla: LOD0 0-50%, LOD1 50-15%, LOD2 15-2%, Culled (görünmez) 2-0%.
  5. Sahnede kameradan uzaklaştıkça mesh otomatik değişir.

Aşırı detaylı asset'ler (markete satılan AAA asset'leri) zaten LOD'lu gelir. Kendi modellerinde Mesh Decimator gibi Unity asset'leri ile otomatik LOD üretebilirsin.

26.8

👁 Occlusion Culling — Görünmeyeni Çizme

Frustum Culling (kamera dışı) ve Back-face Culling (arka yüz) Unity'de varsayılan. Ama duvarın arkasındaki düşman kameranın görüş alanında ama duvarla engellenmiş — bunu da çizmemek için Occlusion Culling kullanılır.

  1. Çevre objelerini Static ☑ (Occluder Static ve Occludee Static).
  2. Window → Rendering → Occlusion Culling.
  3. "Bake" → Unity sahneyi tarayıp hangi açıdan ne görünür hesaplar (5-30 dk).
  4. Runtime'da kamera kendi konumuna bakarak gizli olanları çizmeyi atlar.

Kapalı koridorlu sahnelerde (FPS, RPG dungeon) %50+ GPU kazancı verebilir.

26.9

🖼 Texture Streaming & Compression

Yüksek çözünürlüklü texture'lar bellek dolduran ana suçlulardır. 4K 4096x4096 PNG = 64 MB; bir sahnede 50 tane = 3 GB.

Texture Streaming

  • Quality Settings → Texture Streaming ☑.
  • Uzaktaki objelerin texture'ları düşük mipmap (1/4, 1/8) yüklenir.
  • Yaklaşınca otomatik yüksek mipmap'e geçer.
  • Bellek tüketimi %60-80 azalır.

Texture Compression

  • Texture Inspector → Platform tab → Override for Android/iOS/PC.
  • Android: ASTC 6x6 (kalite-boyut dengesi).
  • iOS: ASTC 6x6 veya PVRTC.
  • PC: BC7 (kalite) veya DXT5 (yaygın).
  • Mipmap Streaming ☑ — runtime'da yüklenmesin yüksek çözünürlük.
26.10

⏱ Async Scene Loading

SceneManager.LoadScene sahneyi senkron yükler, oyun donar. LoadSceneAsync arka planda yükler, oyuncuya loading bar göstersin. 

private IEnumerator LoadSceneRoutine(string sceneName)
{
    var op = SceneManager.LoadSceneAsync(sceneName);
    op.allowSceneActivation = false;

    while (op.progress < 0.9f)
    {
        float progress = Mathf.Clamp01(op.progress / 0.9f);
        loadingBar.fillAmount = progress;
        loadingText.text = $"{progress * 100:F0}%";
        yield return null;
    }

    // Tam yüklendi; tuşa basınca aktif et
    pressAnyKeyText.SetActive(true);
    while (!Input.anyKeyDown) yield return null;
    op.allowSceneActivation = true;
}

Progress 0-0.9 yükleme, 0.9-1.0 sahne aktivasyonu (eski sahne unload, yeni sahne enable). allowSceneActivation = false ile 0.9'da bekletip kontrolü oyuncuya bırakabilirsin.

26.11

🗑 GC.Alloc — Garbage Collector Düşmanı

.NET'in Garbage Collector'ı çalıştığında oyun 1-50 ms donar. Sık çalışırsa hissedilir stutter olur. GC'yi tetikleyen ana sebep: her frame yeni allocation.

Yaygın Suçlular

  • new Vector3(x,y,z) bir struct, alloc yok. new GameObject() heap alloc.
  • Update'te string s = "score: " + score; → her frame string concat = yeni string allocation. StringBuilder kullan.
  • Find, FindObjectsOfType her çağrıda alloc + yavaş. Awake'te bir kere cache'le.
  • GetComponent Update'te → cache'le.
  • LINQ Update'te → for döngüsü kullan.
  • Closure (lambda capture) Update'te → değişkeni dışarı al.
  • foreach bazı koleksiyonlarda enumerator alloc → for'a çevir.

Profiler'da GC Görünür

Profiler → CPU → Hierarchy → GC Alloc sütununu sırala. Sıfır olmayan satırlar Update'te alloc yapıyor. Sıraya göre fix.

💡 Incremental GC: Player Settings → Other → Use Incremental GC ☑. GC işini birkaç frame'e böler, tek bir spike yerine küçük artışlar.
26.12

🎚 Quality Settings & Platform Hedefi

Edit → Project Settings → Quality. Hazır kalite seviyeleri: Very Low, Low, Medium, High, Ultra. Her platform varsayılan bir seviye seçer.

  • Pixel Light Count — Mobil 1, Konsol 4, PC 8.
  • Texture Quality — Quarter Res (mobil), Full Res (PC).
  • Anisotropic Filtering — Forced On PC, Per Texture Mobil.
  • Anti Aliasing — Mobil 2x, PC 4x veya MSAA off + URP'de FXAA/SMAA.
  • Soft Particles — Pahalı, mobil'de kapat.
  • Realtime Reflection Probes — Yansıma anlık güncelleme; sadece üst seviye kalite.
  • Shadow Distance — Mobil 30m, PC 100m.
  • VSync Count — Don't Sync mobile, Every V Blank PC.

Oyunun "Düşük / Orta / Yüksek / Ultra" ayar menüsü hep bu kalite presetlerine bağlanır: 

public void OnQualityChanged(int level)
{
    QualitySettings.SetQualityLevel(level, applyExpensiveChanges: true);
}
26.13

💾 Memory Profiler

Standart Profiler memory ana hatlarını gösterir. Detay için Memory Profiler paketini Package Manager'dan yükle.

  • Take Snapshot ile bellek görüntüsü al.
  • Snapshot'lar arasında karşılaştır: "Level 1'e geçince hangi yeni asset'ler yüklendi?"
  • Memory leaks (referansı kalmış unload edilemeyen asset'ler) bulunur.
  • Texture, mesh, audio breakdown'u en büyük tüketenleri gösterir.

Resources.UnloadUnusedAssets() bir sahne geçişinde manuel temizlik. System.GC.Collect() .NET çöp toplama (bilinçli kullan, oyun durur).

✅ Optimizasyon Kontrol Listesi

  • ☐ Profiler ile darboğazı buldun mu? (CPU mu GPU mu Memory mi?)
  • ☐ SRP Batcher açık mı?
  • ☐ GPU Instancing materyallerde ☑?
  • ☐ Static objeler Batching Static ☑?
  • ☐ Sık spawn olan objeler Object Pool kullanıyor mu?
  • ☐ Update'te GetComponent / Find var mı? Cache'le.
  • ☐ Texture'lar platform için compress edildi mi?
  • ☐ Audio dosyaları doğru import ayarında mı? (Müzik Streaming, kısa efekt Decompressed)
  • ☐ Light count makul mü? Static lighting bake edildi mi?
  • ☐ LOD Group ekledin mi karmaşık mesh'lere?
  • ☐ Occlusion Culling bake edildi mi?
  • ☐ Quality Settings platform için doğru mu?
  • ☐ Async scene loading kullanılıyor mu?
  • ☐ Mobile'da target frame rate set edilmiş mi? (Application.targetFrameRate = 60)
  • ☐ Incremental GC açık mı?
  • ☐ Build Report ile en büyük asset'ler tespit edildi mi?
🚀 Bölüm Kontrol Noktası: Önce ölç (Profiler), sonra optimize. SRP Batcher + GPU Instancing draw call düşürür. Object Pool sürekli oluşan objeler için. Job System + Burst ağır hesaplamalar için. LOD + Occlusion Culling + Texture Streaming GPU/memory için. GC.Alloc'u Update'ten temizle. Quality Settings platform için ayarla.

🎯 Pratik Projeler

Proje 1: Profiler ile Bottleneck Bul + Optimize Et

Hedef: Mevcut bir prototipinde Profiler ile darboğazı bul, en az 3 iyileştirme yap, FPS'i ölç.

  1. Geliştirme build'i al, Profiler ile bağlan.
  2. 30 saniye oyna, sapan frame'leri kaydet.
  3. Hierarchy görünümünde Self ms'i en yüksek 5 metot listele.
  4. Bunların kaynağını bul: Update'te GetComponent? LINQ? string concat? Instantiate?
  5. İlk 3'ünü düzelt.
  6. Aynı build alanını ölç, before/after FPS'i karşılaştır.

Proje 2: Object Pool ile Mermi/Düşman Sistemi

Hedef: Sürekli Instantiate/Destroy yapan bir spawner'ı pool'a çevir, GC.Alloc'u görsel ölç.

  1. Saniyede 30 mermi spawn eden eski spawner'ı al.
  2. Profiler → CPU → GC Alloc sütununa bak: muhtemelen 500-2000 byte/frame.
  3. UnityEngine.Pool.ObjectPool ile rewrite et.
  4. Bullet kendi pool'a Release etsin (Init'ten pool reference al).
  5. Aynı sahneyi koş; GC Alloc'u tekrar ölç.
  6. Hedef: 0 byte/frame.

Proje 3: Job System ile 50.000 Parçacık

Hedef: Single-threaded'ta donduran bir partikül sistemini Job + Burst ile paralelleştir.

  1. 50000 GameObject'in pozisyonunu her frame Update'te güncellemek FPS'i 5'e düşürür.
  2. NativeArray<float3> positions, velocities.
  3. [BurstCompile] struct MoveJob : IJobFor — Execute(int i): positions[i] += velocities[i] * dt.
  4. job.ScheduleParallel(N, 64, default).Complete().
  5. Sonra positions'ı bir Compute Buffer'a yaz, instanced mesh ile çiz (Graphics.DrawMeshInstancedIndirect).
  6. FPS karşılaştır: Single-threaded vs Job vs Job+Burst.
